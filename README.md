# 3D FEM Viewer — README

Interactive WebGL + WebAssembly visualiser for finite element solutions.  
PDE solved: **−div(η ∇u) + λu = f** on Ω = [−a, a] × [−b, b]

---

## Project Structure

```
3d-viewer/
├── index.html        — single-page app (WebGL canvas + control panel + code viewer)
├── build.sh          — Emscripten build script
├── app.js            — compiled WASM output (generated by build.sh)
├── app.wasm          — (generated)
├── README.md         — this file
└── src/
    ├── Camera.h      — Vec3, Mat4, spherical-orbit Camera (Z-up)
    └── source.cpp    — WebGL rendering + FEM solver
```

## Building

Requires [Emscripten](https://emscripten.org/) (`emcc` on PATH):

```bash
cd 3d-viewer
bash build.sh
```

Then open `index.html` in any modern browser (Chrome, Firefox, Edge).

## Original C++ → WASM Adaptation: Mapping Table

The original code is a standalone C++14 FEM library.  
The WASM version runs in the browser via Emscripten and WebGL.  
**The mathematics is identical; only the infrastructure changed.**

### Data Structures

| Original (`source/fem_original.h`) | WASM (`src/source.cpp`) | Reason for change |
|------------------------------------|-------------------------|-------------------|
| `struct Noeud { double x, y; }` | `FEMState::Xc[], Yc[]` (plain arrays) | Simpler layout for pre-computed mesh |
| `struct Triangle { int i, j, k; }` | `std::vector<std::array<int,3>> FEMState::tri` | Same indexing, different wrapper |
| `struct ETA` (std::function) | `EM_JS double js_eval_eta(x,y)` | `std::function` cannot cross the C++↔JS boundary; `EM_JS` bridges to user-typed JavaScript |
| `struct RHSF` (std::function) | `EM_JS double js_eval_f(x,y)` | Same reason |
| `struct VhElement` | Inline coordinate lookup per triangle | Avoids extra allocation inside each `matvec` call |

### Mesh Generation

| Original | WASM | Notes |
|----------|------|-------|
| `maillageTR(a,b,N,M, noeuds, triangles)` | `FEMState::build(N,M,a,b,lam,dir)` | Loop structure and index formulas are character-for-character identical |

The mesh is pre-computed once and stored in the global `g_fem` struct, then reused for every matrix-vector call inside BiCGSTAB.

### Element Routines

| Original | WASM | Change |
|----------|------|--------|
| `CalcMatBT(el, b00,b01,b10,b11)` | `calcBT(tx, ty, …)` | Same formula, receives raw arrays instead of `VhElement` |
| `integ_eta(el, eta)` | `integEta(tx, ty)` | Same midpoint quadrature rule |
| `DiffTerm(el, eta, K[6])` | `diffTerm(tx, ty, K[6])` | **Mathematically identical** |
| `ReacTerm(el, lam, Ml[6])` | `reacTerm(tx, ty, lam, Ml[6])` | **Mathematically identical** |
| `symPacked(i,j)` | `symIdx(i,j)` | Same 6-entry upper-triangle packing |

### Matrix-Vector Product

| Original | WASM | Notes |
|----------|------|-------|
| `matvec(elements, eta, lam, V, W, …)` | `fem_matvec(V)` — reads `g_fem` global | Same assembly loop |
| `calcAV(elements, V, W, …)` | Not ported | Only used for `normL2`; norms are shown via `umin/umax` on the JS side |

### BiCGSTAB Solver — ⚠ Critical Bug Fixed in v13

Both versions implement the same BiCGSTAB algorithm (Van der Vorst 1992).  
The WASM **v12** had a subtle bug that the original code does **not** have:

The p-update step must use the **old A·p** from the previous iteration:

```
p[i] = r[i] + beta * ( p[i]  −  omega * OLD_Ap[i] )
```

In the original `bicgstab()`, variable `v` naturally holds `old A·p` at that point in the loop — **correct**.

In the WASM v12 rewrite, variable `T` (which held `old A·s`, not `old A·p`) was accidentally used:

```cpp
// WRONG (v12):
p[i] = r[i] + beta*(p[i] - omega * T[i]);     // T = old A*s  ← BUG

// CORRECT (v13):
p[i] = r[i] + beta*(p[i] - omega * v_vec[i]); // v_vec = old A*p ← FIXED
```

**Effect:** v12 gave wrong solutions for non-constant η.  
**Fix in v13:** `v_vec` is declared outside the loop and persists, always holding the most recently computed A·p before being overwritten.

### Boundary Conditions

| Original | WASM |
|----------|------|
| `onBnd[]` bool vector passed as argument | `FEMState::onBoundary(s)` computed inline |
| Caller ensures λ > 0 for Neumann | WASM enforces `λ ≥ 1e-6` automatically |

### Norms

`normL2` and `normL2Grad` are not ported into the WASM C++ solver.  
After `solvePDE()` returns, the JS side reads `umin/umax` from the solution vector and displays them in the colorbar.  
The same formulas can be added if needed.

## PDE Problem

```
−∂/∂x(η ∂u/∂x) − ∂/∂y(η ∂u/∂y) + λu = f    on Ω = [−a,a] × [−b,b]
```

- **f(x,y)**: right-hand side (user-supplied, supports `sin`, `cos`, `exp`, `sqrt`, `pi`, …)
- **η(x,y)**: diffusion coefficient
- **λ**: reaction coefficient
- Boundary conditions: Dirichlet (u = 0) or Neumann (∂ₙu = 0)

Discretised with P1 triangular elements (linear hat functions).  
Solved iteratively with BiCGSTAB (no explicit sparse matrix stored).
